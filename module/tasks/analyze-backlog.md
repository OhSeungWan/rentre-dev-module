# Quick Analyze Backlog Task

단일 백로그 아이템을 빠르게 분석하는 태스크

## 용도

이 태스크는 PM 에이전트의 `*analyze` 명령으로 실행됩니다.
백로그를 분해하기 전 빠른 분석이 필요할 때 사용합니다.

**사용 시나리오:**

- 백로그 복잡도 빠른 평가
- 서브태스크 수 예상
- 잠재적 리스크 식별
- 분해 전 사전 검토

## 실행 단계

### 1. 백로그 입력

사용자로부터 백로그 아이템 정보를 입력받습니다:

- 제목 (필수)
- 설명/요구사항 (필수)
- 수용 기준 (선택)
- 백로그 유형 (선택: Epic/Story/Task/Bug)

### 2. 빠른 분석 수행

#### 2.1 요구사항 분석

**기능 요구사항 추출:**

- 사용자 액션 관련 요구사항 식별
- CRUD 패턴 감지
- 비즈니스 로직 요구사항

**비기능 요구사항 식별:**

- 성능 요구사항 (응답 시간, 처리량)
- 보안 요구사항 (인증, 권한)
- 접근성 요구사항
- 호환성 요구사항

**모호한 부분 표시:**

- 불명확한 표현 하이라이트
- 누락된 정보 식별
- 가정이 필요한 부분

#### 2.2 복잡도 평가

**작업량 평가 (T-Shirt Sizing):**

| 사이즈 | 설명                     | 예상 시간 |
| ------ | ------------------------ | --------- |
| S      | 단순 변경, 1-2개 파일    | 2-4시간   |
| M      | 중간 복잡도, 3-5개 파일  | 4-8시간   |
| L      | 높은 복잡도, 5-10개 파일 | 1-2일     |
| XL     | 매우 복잡, 10개+ 파일    | 3-5일     |

**기술적 복잡도 평가:**

- 신규 기술/라이브러리 필요 여부
- 기존 코드 수정 범위
- API/데이터베이스 변경 필요 여부
- 외부 의존성

**의존성 파악:**

- 선행 작업 필요 여부
- 다른 백로그와의 연관성
- 외부 시스템 연동

#### 2.3 코드 영향 분석 (키워드 기반)

**영향받는 영역 예측:**

- 백로그 키워드에서 관련 코드 영역 추론
- 프론트엔드/백엔드/DB 구분
- 예상 변경 파일 유형

**잠재적 충돌 영역:**

- 자주 변경되는 파일 예상
- 병합 충돌 가능성

## 다음 단계 연계

분석 완료 후 사용자에게 다음 옵션 제시:

1. **`*prepare`** - 상세 백로그 준비 (요구사항 구조화, 컨텍스트 수집)
2. **`*decompose`** - 바로 백로그 분해 진행
3. **`*analyze-code`** - 코드베이스 분석 먼저 진행
4. **완료** - 분석만 하고 종료

## 출력 형식

- 분석 결과를 화면에 마크다운 형식으로 표시
- 필요시 prepare-backlog 또는 decompose-backlog 워크플로우로 연계
- 분석 결과는 별도 파일로 저장하지 않음 (빠른 분석 목적)
